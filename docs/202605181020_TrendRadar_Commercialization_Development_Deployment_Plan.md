# TrendRadar 品牌舆情与营销机会雷达商业化作战计划

文档版本: v1.0  
编写日期: 2026-05-18  
适用项目: AIYXDATA-TRADAR / TrendRadar  
当前代码版本: 1.01.0511  
目标读者: 项目负责人、开发团队、商业化负责人、交付实施团队

商业目标: 先实现年收入 500 万人民币

## 1. 执行摘要

TrendRadar 当前已经具备一个可运行的“热点抓取 + RSS 聚合 + 关键词过滤 + AI 分析 + 多渠道推送 + HTML 报告 + MCP 查询”的基础能力。它不是一个普通 RSS 阅读器，而是一个轻量级信息情报管线。

本阶段最适合的商业化方向，不是做泛化的企业级全网舆情平台，也不是同时铺开投研、城市、电商、政策等多个方向，而是聚焦一个更容易成交、更容易交付、更能形成案例的场景：

> 品牌舆情与营销机会雷达。

它的核心价值不是“监控全网”，而是每天帮助品牌团队、公关团队、市场团队和营销代理商回答四个问题：

- 今天有没有品牌风险？
- 竞品在发生什么？
- 有没有可以借势的热点和内容选题？
- 市场团队今天应该做什么动作？

反常识判断:

- 商业化时不要强调“我能抓很多信息”，而要强调“我能替你少看信息”。
- 不要先做大屏，先做能每天被打开的早报、晚报和异常提醒。
- 不要让用户从零配置关键词，应该出售已经调好的品牌模板。
- 不要优先打大型集团，先打中小品牌、公关公司、营销代理商和区域消费品牌。
- 不要和 Meltwater、Brandwatch、Mention 正面对比“覆盖范围”，应该对比“部署成本、中文热榜、品牌场景模板、私有可控、AI 助理化”。

年收入 500 万不是靠低价工具订阅完成，而应通过“软件 + 模板 + 人工增强 + 代理商批量交付 + 私有部署”的混合模式实现。

### 1.1 500 万年收入目标拆解

建议采用以下收入结构:

| 收入来源 | 数量 | 单价 | 年收入 |
| --- | ---: | ---: | ---: |
| 增长雷达版品牌客户 | 20 个 | 4999 元/月 | 约 120 万 |
| 代理商版客户 | 5 个 | 3 万元/月 | 180 万 |
| 企业私有部署 | 10 个 | 15 万元/年 | 150 万 |
| 定制报告、品牌模板、人工增强服务 | 若干 | 项目制 | 50 万 |
| 合计 | - | - | 约 500 万 |

这个拆解的关键不是客户数量多，而是客单价足够高、交付足够标准化。第一年不追求上千个用户，优先拿下 30-50 个高质量付费客户和 5 个代理商/服务商渠道。

### 1.2 第一阶段只做什么

第一阶段只做一个主产品:

> 品牌每日舆情与营销机会助理。

标准交付物:

- 每日舆情日报。
- 重大负面风险提醒。
- 每周品牌/竞品/营销机会周报。
- 竞品动作追踪。
- 可借势热点和内容选题建议。
- 原始来源链接和证据链。
- 人工校对版报告，面向高客单价客户。

### 1.3 第一阶段不做什么

第一阶段不做:

- 全网舆情大平台。
- 复杂 BI 大屏。
- 面向所有行业的通用情报市场。
- App。
- 复杂支付系统。
- 重型多租户 SaaS。
- 和大型舆情平台拼数据覆盖。

这些能力可以后续补，但不应该阻塞第一批收入。

## 2. 当前项目现状

### 2.1 当前定位

项目 README 将 AIYXDATA-TRADAR 定义为“AI 驱动的全渠道热点情报助手”。实际代码中的定位可以更精确地表达为:

> 面向个人和小团队的自动化热点情报系统。它从热榜平台和 RSS 源采集信息，通过关键词与标签规则筛选，再由 AI 生成研判，最后以 HTML 报告和协作工具消息的形式分发。

### 2.2 当前已具备的核心能力

当前项目已经具备以下基础能力:

1. 热榜数据抓取
   - 支持今日头条、百度热搜、华尔街见闻、澎湃新闻、Bilibili 热搜、财联社、凤凰网、贴吧、微博、抖音、知乎等。
   - 默认通过 `https://newsnow.busiyi.world/api/s` 获取热榜数据。
   - 支持请求间隔、重试、代理。

2. RSS/Atom 聚合
   - 支持自定义 RSS/Atom 源。
   - 当前配置包含 Hacker News、36KR、FT 中文网、36 氪、虎嗅、经济学人相关源、疯投圈等。
   - 支持新鲜度过滤，例如只推送最近 N 天的文章。

3. 关键词与标签系统
   - 使用 `config/frequency_words.txt` 配置关注词。
   - 支持普通关键词、正则表达式、别名、组别名、必须词、排除词、显示条数限制和 `@tags` 分类标签。
   - 支持全局过滤词。

4. 三种报告模式
   - `incremental`: 增量模式，只关注新增内容，降低重复打扰。
   - `current`: 当前榜单模式，关注当前仍在榜的内容。
   - `daily`: 当日汇总模式，汇总全天匹配内容。

5. AI 分析
   - 通过 LiteLLM 接入模型服务。
   - 支持 OpenAI 兼容接口和多种模型提供商。
   - 当前 AI 分析结果结构包括核心热点、舆论风向、异动弱信号、RSS 洞察、策略建议、独立展示区概括。
   - 支持 AI 分析提示词文件。

6. 多渠道通知
   - 飞书、钉钉、企业微信、Telegram、邮件、ntfy、Bark、Slack、Generic Webhook。
   - 支持多账号配置。
   - 支持内容分批发送。

7. 自动化 HTML 报告
   - 生成历史 HTML 报告。
   - 维护 `output/html/latest/current.html`、`daily.html`、`incremental.html` 等最新报告。
   - 报告包含主题样式、卡片布局、AI 分析、热榜、RSS、新增内容、独立展示区等。

8. 调度系统
   - 使用 `config/timeline.yaml` 控制采集、分析、推送时间。
   - 支持 `always_on`、`morning_evening`、`office_hours`、`night_owl`、`custom` 等预设。
   - 支持“每个时间段只分析/推送一次”的控制。

9. 可视化配置中心
   - 位于 `docs/index.html` 和 `output/config_editor/`。
   - 左侧为 YAML/频率词/时间线源码编辑器。
   - 右侧为可视化配置面板。
   - 支持读取当前配置、加载默认配置、保存方案、应用同步、AI 模型查询。

10. MCP Server
    - 使用 FastMCP。
    - 提供自然语言日期解析、最新新闻查询、趋势话题、RSS 查询、搜索、分析、通知等工具。
    - 可作为 AI Agent 的本地新闻/趋势数据源。

11. 存储
    - 本地 SQLite 数据库存储热榜和 RSS。
    - 输出 HTML、TXT 快照。
    - 支持 S3 兼容远程存储。
    - 支持从远程拉取最近 N 天数据。

12. 部署形态
    - Python 命令行运行。
    - Docker / Docker Compose 运行。
    - GitHub Actions 定时运行。
    - MCP Server 独立运行。

## 3. 当前产品形态与用户体验

### 3.1 当前使用流程

典型使用流程:

1. 用户编辑 `config/config.yaml`，配置平台、RSS、AI Key、通知渠道和存储。
2. 用户编辑 `config/frequency_words.txt`，配置关注关键词和过滤词。
3. 用户编辑 `config/timeline.yaml`，配置采集、分析、推送时间。
4. 程序被 Docker cron、GitHub Actions 或手动命令触发。
5. 程序抓取热榜和 RSS。
6. 程序保存 SQLite 数据和 HTML 报告。
7. 程序按关键词和报告模式进行统计。
8. 程序调用 AI 生成研判。
9. 程序将报告推送到协作工具。
10. 用户查看 HTML 报告或在聊天工具中查看摘要。

### 3.2 当前报告排版

当前 HTML 报告已经具备可读性和一定视觉设计:

- 顶部标题区。
- 版本信息。
- 主题切换能力。
- 数据摘要。
- 新增热点区域。
- 热榜关键词区域。
- RSS 区域。
- 独立展示区。
- AI 分析区。

当前报告更像“信息展示页”，商业化后应改造成“决策报告页”:

- 第一屏直接给今日结论。
- 每个结论都能追溯原始来源。
- 用风险、机会、异常、行动建议替代纯新闻堆叠。
- 每个垂直行业有专属报告结构。

### 3.3 当前配置中心排版

当前 `docs/index.html` 的配置中心布局:

- 顶部导航栏。
- 左右分栏。
- 左侧源码编辑器。
  - `config.yaml`
  - `frequency_words.txt`
  - `timeline.yaml`
- 右侧可视化配置面板。
- 顶部操作按钮:
  - 读取当前配置
  - 锁定编辑
  - 加载默认设置
  - 提取方案
  - 保存为方案
  - 复制配置
  - APPLY 应用同步
- 支持 AI 模型连接测试与模型列表获取。

商业化后配置中心不应继续作为主入口。主入口应该变成:

- 项目/品牌模板列表。
- 新建监控项目向导。
- 报告和提醒历史。
- 关键词和数据源只作为高级设置。

## 4. 当前商业化缺口

### 4.1 产品定位缺口

当前项目讲的是“我能聚合热点”，商业客户需要的是“我每天应该关注什么、采取什么行动”。

需要补齐:

- 明确目标用户。
- 明确垂直场景。
- 明确报告模板。
- 明确信号评分。
- 明确交付结果。

### 4.2 安全缺口

当前不适合直接商用的关键原因:

- 配置文件中存在疑似真实 AI API Key，且 `config/config.yaml` 被 Git 跟踪。
- 配置编辑服务存在读取和写入配置的接口，当前没有看到完整鉴权。
- `Access-Control-Allow-Origin: *` 对公网暴露有风险。
- Webhook、AI Key、S3 Key 等敏感信息缺少统一密钥管理。
- MCP Server 如果暴露 HTTP，需要访问控制和租户隔离。

必须优先完成:

- 轮换已经暴露的密钥。
- 从配置文件中移除真实密钥。
- 使用环境变量或密钥管理器。
- 配置中心加登录和权限。
- 所有写接口加鉴权、CSRF/Token、防重放。
- 日志中脱敏所有 Key、Token、Webhook。

### 4.3 数据源稳定性缺口

当前热榜依赖一个外部 NewsNow API，这是商业化最大单点风险之一。

需要补齐:

- 数据源健康检查。
- 同类源备份。
- 抓取失败自动降级。
- 源质量评分。
- 源授权和使用边界梳理。
- 重要客户的数据源 SLA 说明。

### 4.4 多租户与账号缺口

当前更像单人自部署工具。商业化需要:

- 用户体系。
- 组织体系。
- 项目体系。
- 权限体系。
- 多租户数据隔离。
- 套餐和额度。
- 操作审计。

### 4.5 商业报告缺口

当前报告展示了信息，但还没有充分产品化成“判断”。

需要补齐:

- 风险、机会、异常、行动建议。
- 事实证据链。
- 变化趋势。
- 竞品对比。
- 每周/月度总结。
- 报告导出。
- 报告反馈。

### 4.6 运营与交付缺口

当前缺少:

- 品牌模板库。
- 默认关键词包。
- 演示报告。
- 客户试用流程。
- 部署说明。
- 价格体系。
- 服务条款。
- 数据合规说明。
- 客户成功流程。

## 5. 推荐商业化总方向

### 5.1 不推荐方向

不建议将 TrendRadar 定位为:

- 泛化全网舆情监控平台。
- 企业级社媒大数据分析平台。
- 大而全 BI 大屏。
- 单纯 RSS 阅读器。
- 单纯 AI 新闻总结工具。

这些方向要么竞争极强，要么用户付费意愿低，要么数据源和合规成本过高。

### 5.2 推荐方向

推荐定位:

> 面向商业品牌、公关公司、市场团队和营销代理商的 AI 舆情与营销机会助理。

核心价值:

- 减少信息噪音。
- 发现品牌风险。
- 追踪竞品动作。
- 识别可借势热点和内容选题。
- 给出可行动的营销和公关建议。
- 每天定时推送。
- 可自部署、可私有化、可低成本运行。

核心卖点:

- 中文热榜 + RSS + 品牌关键词模板。
- AI 生成决策摘要，而不是新闻堆叠。
- 飞书/企微/Telegram/邮件原生推送。
- 既做舆情防守，也做营销机会发现。
- 支持私有部署。
- 支持 MCP，后续能作为 AI Agent 的品牌情报记忆库。

## 6. 商业化场景设计: 品牌舆情与营销机会雷达

本阶段商业化只围绕一个主线展开:

> 帮商业品牌、公关团队、市场团队和营销代理商每天自动发现风险、竞品动作和营销机会。

其他方向如投研、城市、电商、政策、MCP 数据服务可以保留为后续扩展，但不进入第一阶段 500 万收入目标的主路径。

### 6.1 主战场: 中小消费品牌舆情雷达

目标客户:

- 新消费品牌。
- 美妆护肤品牌。
- 食品饮料品牌。
- 母婴、宠物、服饰、家居品牌。
- 抖音、小红书、天猫、京东、私域起量中的成长型品牌。

客户痛点:

- 品牌没有专人每天盯热榜、社媒和行业媒体。
- 负面评论、投诉、价格争议、产品质量争议可能突然扩散。
- 竞品爆款和营销打法容易错过。
- 内容团队每天缺选题，营销动作依赖灵感。

产品交付:

- 每日品牌舆情日报。
- 高风险负面提醒。
- 竞品动态追踪。
- 用户关注点变化。
- 可借势热点和内容选题。
- 每周品牌/竞品/营销机会周报。

报告示例:

```text
今日风险等级: 中

主要风险:
- 百度/知乎出现关于“价格争议”的讨论，暂未形成跨平台扩散。
- 建议客服团队关注同类问题，准备统一回应话术。

竞品动态:
- 竞品 A 的新品包装在短视频平台获得讨论，可观察其视觉卖点。

营销机会:
- “春季露营”“轻量出行”相关话题升温，适合结合产品做场景内容。

今日建议动作:
1. 客服团队关注价格相关评论。
2. 内容团队准备露营场景短视频。
3. 市场团队观察竞品 A 的新品反馈。

证据来源:
- 链接 1
- 链接 2
- 链接 3
```

可实现性:

- 当前已有热榜、RSS、关键词、AI 分析、飞书/企微/邮件推送能力。
- 第一版不需要全网覆盖，先覆盖热榜 + RSS + 合法公开源。
- 通过品牌词、竞品词、负面词、行业词、营销机会词即可形成可卖报告。

商业价值:

- 预算来源明确，可归入公关、市场、内容运营、品牌咨询费用。
- 客户能直接感知“节省时间”和“降低风险”。
- 可通过人工增强报告提高客单价。

### 6.2 高优先级渠道: 公关公司与营销代理商

目标客户:

- 中小公关公司。
- 品牌营销代理商。
- 内容代运营公司。
- 本地生活服务商。
- MCN 和品牌内容服务商。

为什么优先:

- 一个代理商手里往往有 5-30 个品牌客户。
- 代理商本来就需要给客户交日报、周报、竞品报告。
- 它们可以把 TrendRadar 的报告包装成自己的服务，提高客户粘性。
- 单个代理商客户的客单价远高于单品牌客户。

产品交付:

- 多品牌项目管理。
- 批量生成日报/周报。
- 客户专属报告 Logo 和模板。
- 代理商内部飞书/企微推送。
- 客户可读的外发版报告。
- 人工校对工作台。

建议报价:

- 10 个品牌以内: 2 万元/月。
- 20 个品牌以内: 3 万元/月。
- 30 个品牌以内: 5 万元/月。
- 超出按品牌数或报告数加收。

500 万目标中的作用:

- 5 个代理商客户，每个 3 万元/月，年收入约 180 万。
- 代理商是最适合快速放大收入的客户类型。

### 6.3 区域连锁与本地生活品牌

目标客户:

- 餐饮连锁。
- 商场和商业地产。
- 文旅项目。
- 展会和活动品牌。
- 本地生活服务品牌。
- 城市区域品牌。

客户痛点:

- 门店评价、食品安全、服务争议、排队体验容易形成舆情。
- 本地活动、演唱会、展览、市集、天气、交通会影响经营。
- 竞品门店活动和联名营销需要持续关注。

产品交付:

- 城市本地热点。
- 门店风险提醒。
- 活动机会提醒。
- 竞品门店动作。
- 本地内容选题。

可实现性:

- 先做上海，再复制北京、深圳、杭州、成都。

建议报价:

- 单城市单品牌: 2999-6999 元/月。
- 多城市连锁: 1 万-3 万元/月。
- 商业地产/文旅项目: 项目制或年费制。

### 6.4 创始人 IP 与个人品牌

目标客户:

- 知识付费品牌。
- 投资人/财经 KOL。
- 教育培训品牌。
- 医生、律师、咨询顾问等专业服务 IP。

客户痛点:

- 个人品牌声誉风险更集中。
- 一个错误话题可能快速影响信任。
- 创始人需要知道行业热点，及时输出观点。

产品交付:

- 个人品牌声誉监控。
- 负面讨论提醒。
- 行业热点观点建议。
- 适合发布的内容选题。
- 竞品/同行观点追踪。

建议报价:

- 个人 IP 基础版: 1999-4999 元/月。
- 高影响力 IP 人工增强版: 1 万-3 万元/月。

### 6.5 品牌营销机会雷达

这个场景是从“舆情防守”走向“营销增长”的关键，决定产品能否从成本项变成增长工具。

要发现的机会:

- 可借势热点。
- 节日和事件营销机会。
- 竞品爆款内容结构。
- 用户新需求和新吐槽。
- 行业情绪变化。
- 平台流行表达。
- 品类新趋势。

报告输出:

```text
今日营销机会等级: 高

机会 1: “露营轻食”话题升温
- 适合品牌: 食品饮料、户外用品、轻量厨具
- 建议内容: 3 个短视频脚本方向
- 风险提醒: 避免过度蹭灾害/事故类热点

机会 2: 竞品 A 新包装引发讨论
- 可学习点: 视觉反差强、评论区互动高
- 可反击点: 用户质疑价格偏高
- 建议动作: 做一条“同场景更实用”的对比内容
```

为什么有商业价值:

- 负面监控只能证明“避免损失”，营销机会能证明“创造增长”。
- 市场团队更愿意为能产出选题和动作建议的工具付费。
- 代理商可以直接把这些建议转化成内容方案和客户提案。

### 6.6 第一阶段暂缓的方向

以下方向有潜力，但不作为 500 万目标的第一主线:

- AI/机器人/半导体投研雷达。
- 政策与监管风险雷达。
- B2B 销售触发器。
- 电商品类选品雷达。
- 通用 MCP 新闻数据服务。

暂缓原因:

- 销售对象分散。
- 报告模板差异大。
- 数据源和判断标准不同。
- 会拉长产品聚焦周期。

后续可以在品牌主线验证成功后，把已有能力复用到这些方向。

## 10. 部署方案

### 10.1 当前 Docker 部署

当前 `docker-compose.yml`:

```yaml
services:
  aiyxdata_tradar:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: aiyxdata_tradar
    ports:
      - "127.0.0.1:8084:8080"
    volumes:
      - ./data:/app/data
      - ./config:/app/config
    environment:
      - PYTHONUNBUFFERED=1
    restart: unless-stopped
```

注意:

- 当前只映射了 `data` 和 `config`。
- 商业部署建议显式映射 `output`，否则报告和数据库可能留在容器内。

建议改造:

```yaml
services:
  trendradar:
    image: trendradar:latest
    container_name: trendradar
    ports:
      - "127.0.0.1:8084:8080"
    volumes:
      - ./config:/app/config
      - ./output:/app/output
      - ./data:/app/data
      - ./logs:/app/logs
    env_file:
      - .env
    restart: unless-stopped
```

### 10.2 私有部署推荐架构

```text
Internet
   |
   v
Nginx/Caddy HTTPS 443
   |
   +-- /              -> TrendRadar Web 8080
   +-- /config_editor -> Auth-protected Config UI
   +-- /api           -> API Server
   +-- /mcp           -> MCP Server, optional, internal first
```

推荐端口:

- `443`: HTTPS 对外。
- `8080`: TrendRadar Web 内部端口。
- `8084`: 本机映射端口。
- `8090`: MCP HTTP 服务预留端口。
- `5432`: PostgreSQL，SaaS 或企业版可选，内网。
- `6379`: Redis/Queue，可选，内网。

- 飞书/企微/邮件推送模板。
- 人工校对流程。

验收标准:

- 30 分钟内能为一个品牌配置项目。
- 当天能生成第一份日报。
- 日报包含风险、竞品、机会、建议动作和证据链。
- 可连续 14 天为试点客户稳定推送。

### 阶段 2: 付费试点与人工增强交付

周期: 4-8 周

目标:

- 找到愿意付费的真实客户，验证报告价值。

销售动作:

- 找 20 个试点客户，不免费，低价试用。

- 每天自动日报。

优先客户:

- 新消费品牌。
- 餐饮连锁。
- 本地生活品牌。
- 公关公司。
- 营销代理商。
- 创始人 IP。

验收标准:

- 至少 20 个试点意向。
- 至少 10 个付费试点。
- 至少 5 个客户愿意续费。
- 至少 3 个客户愿意授权形成匿名案例。

### 阶段 3: 标准套餐与代理商版本

任务:

## 14. 风险与应对

### 14.1 数据源风险

风险:

- 热榜 API 不稳定。
- RSS 源失效。
- 数据源授权不清。

应对:

- 建立数据源注册和健康检查。
- 对核心源做备份。
- 对商业客户明确数据范围。
- 优先使用 RSS、公开 API、授权数据和客户自有数据。

### 14.2 AI 成本风险

风险:

- 频繁 AI 分析导致成本不可控。
- 长上下文报告成本高。

应对:

- 限制每次分析新闻数量。
- 增加分层摘要。
- 使用便宜模型做初筛，高级模型做总结。
- 套餐限制 AI 调用次数。

### 14.3 报告质量风险

风险:

- AI 胡编。
- 总结空泛。
- 用户不信任。

应对:

- 所有结论附来源。
- 明确事实与推测。
- 引入评分和证据链。
- 通过反馈持续优化。
- 关键客户提供人工校对版。

### 14.4 安全风险

风险:

- Key 泄露。
- 配置被篡改。
- Webhook 被滥用。

应对:

- 鉴权。
- 加密存储。
- 日志脱敏。
- 审计。
- 限流。
- 私有部署默认内网访问。

### 14.5 竞争风险

风险:

- 大平台功能更多、数据更全。

应对:

- 避免拼全网覆盖。
- 主打垂直模板、本地中文热榜、低成本私有部署、AI 助理化。
- 用“行业判断”而不是“数据规模”形成差异。

## 17. 与竞品的差异化

市场上已有成熟产品覆盖社交聆听、媒体监控、竞品分析、实时预警和报告。公开资料显示:

- Meltwater 强调跨社媒、新闻、论坛等渠道的品牌智能、风险发现、竞品跟踪和报告。
- Brandwatch Listen 强调实时发现趋势、品牌/产品/竞品对话、情绪分析和 AI 智能提醒。
- Mention 提供社交聆听、品牌监控、报告和社媒管理。
- Grand View Research 预计美国社交媒体聆听市场 2025-2030 年有持续增长，公开页面显示美国市场 2024 年收入约 19 亿美元，2030 年预计约 39 亿美元，2025-2030 CAGR 约 13%。

TrendRadar 不应直接竞争“谁覆盖更全”。更好的差异化:

- 更适合中文热榜和中文专业场景。
- 更适合自部署和私有化。
- 更适合小团队预算。
- 更适合“日报/周报/异常提醒”的轻工作流。
- 更适合品牌、公关、营销代理商快速交付。
- 更强调营销机会，而不是只做风险监控。
- 更容易被定制为品牌模板。
- 后续可扩展为 AI Agent 调用。

## 18. 最终建议

TrendRadar 已经具备品牌舆情与营销机会雷达所需的技术雏形，但还缺“品牌产品包装、营销机会判断、数据质量、安全底座和商业交付能力”。

最优路径:

1. 先修安全。
2. 只做品牌舆情与营销机会雷达这一条主线。
3. 用 10 个付费试点客户验证报告价值。
4. 把有效品牌模板固化为收费产品。
5. 同时保留开源自部署作为获客入口。
6. 用代理商版、私有部署和人工增强服务承接高客单价客户。
7. 以 20 个增长雷达客户、5 个代理商客户、10 个私有部署客户作为第一年 500 万目标模型。

一句话产品方向:

> TrendRadar 不应该只是热点聚合器，而应该成为品牌团队每天自动工作的 AI 舆情与营销机会助理。


商业化后建议每个品牌模板提供专属内容区域:

- 品牌舆情日报: 风险等级、负面线索、传播源、回应建议、证据链。
- 竞品周报: 竞品动作、竞品内容打法、用户反馈、可学习点、可反击点。
- 营销机会日报: 可借势热点、选题建议、标题方向、平台建议、风险提醒。
- 代理商外发报告: 客户摘要、建议动作、可外发证据链、人工校对状态。
